Project 7 FAQ

  1. Do I have to worry about this warning from Visual C++? 

    	warning C26495: Variable 'Arena::m_tooters' is uninitialized. Always initialize a member variable (type.6).

    If you're following the convention that m_nTooters always represents how many pointers in m_tooters (at positions 0 through m_nTooters-1) have well-defined values and that you access only those elements, then no. The compiler is not smart enough to recognize that you're doing that (and thus that there's no need for your constructor to initialize all the array elements), so it pessimistically assumes you might access one of the elements before it's been given a well-defined value and warns you.

  2. I can't figure out what to pass as the first argument to the Tooter constructor. I know it's supposed to be a pointer to the City the Tooter is being added to.

    City::addTooter needs to create a new Tooter. City::addTooter is a member function of the City class. How does a member function of the City class talk about the City object it's operating on? If you answer this, you'll know what to do. (Why does that last sentence remind me of "Speak, friend, and enter" from The Fellowship of the Ring?)

  3. My program seems to work, but crashes when it quits. What's happening?

    At the start of your City's destructor code, say 

            cerr << "Entering City destructor" << endl;

    and at the end, say 

            cerr << "Leaving City destructor" << endl;

    Is your destructor causing the crash? One possibility is that your destructor code itself is incorrect, but it's more likely that your destructor code is fine, but earlier in the program you were careless about managing your array of Tooter pointers, especially when a Tooter was destroyed.

    Learning how to interpret sanitizer messages may well make the task of identifying where in your code something is going wrong easier.

    Drawing a picture of what the m_tooters array looks like, carefully hand-simulate your City::moveTooters code to see what happens if a couple of Tooters are converted during a turn. Do you leave the m_tooters array in a reasonable state with an appropriate m_nTooters value?

  4. How can I check for memory leaks?

    If you run your program under g31 and can play the game long enough to convert, say, ten tooters and then quit the game, that should cause enough allocations and deallocations so that if you have a memory leak because you mismanaged your pointers, you will be told about it with a long message starting LeakSanitizer: detected memory leaks
